§0

System layers — the whole stack

Six layers. Two of them — ingestion (knowledge in) and the investigation runtime (reasoning over it) — are distinct flows and must not be conflated. Cross-cutting facilities (provenance, governance, cache) span all layers.

FIG 0 · Layers 2 and 5 are separate flows. Ingestion writes the graph once; the runtime reads/derives over it and writes only into a working box. Dashed arrows = runtime ⇄ engines ⇄ substrate read/write.
§1

Ingestion pipeline — knowledge in

Per-document, left to right. The data shape changes at every stage (annotated on the arrows). Embedding happens once, at the front, and is reused downstream — it is never recomputed per level or per query. Ingestion is essentially autonomous: low-confidence outputs are flagged provisional and deferred to the runtime review queue rather than blocking.

§3

Data model — the node/edge layers

The graph is layered from raw text upward. The lower layers are perception (what the text says); the middle are epistemic content (facts & conclusions, adjudicated by evidence); the top is the intentional layer (the question being answered). Everything is contained in a Box (tier + provenance unit).

FIG 3 · Data model. Perception → epistemic content → intentional. Observations ground; conclusions are re-derived, never inherited from a source.
§4

Reasoning core — propagation & adjudication data flow

When a fact is added or retracted, belief flows through two algebraically-distinct layers and then adjudication. Layer A gates Layer B: only well-founded nodes get a confidence. Cyclic/oscillating regions are bounded and surfaced, not silently re-iterated.

FIG 4 · The split is forced by algebra: retraction needs subtraction, confidence needs idempotence; one structure can't have both.
§5

Investigation runtime — the loop & where the human is required

The loop is steered by the Task. Three points require a human (amber); the rest are autonomous. The expert never edits state directly — they act through framing, the review queue, soft overrides, and the promotion gate, and the engine re-derives.

FIG 5 · ◆ = human required. Frame (input), Triage (review queue), Revise (override / new evidence), and the optional Promotion gate. Everything else is autonomous.

§7

Input & output layers

INPUT

  • Case documents — the evidence of this investigation (tens of dense docs). Per-investigation ingest.
  • Static reference corpus — domain/industry knowledge. Ingested once, reused read-only.
  • Domain pack(s) — taxonomy, entity-type ontology, rules, interest model, reference hypothesis set.
  • Task / question — the framing input that steers the loop.
  • Expert actions — review-queue judgements, soft overrides, promotion approvals.

OUTPUT

  • Initial conclusion — the Task's addressing hypotheses + state + acceptability (banded true/plausible/implausible/false). Not a verdict.
  • Evidential subgraph — the SUPPORTS/REFUTES structure behind each conclusion.
  • Provenance trail — every node resolvable to its source span (clearance-permitting).
  • Review queue — the ranked, budgeted set of items most worth a human's time.
  • Audit log — append-only record of every operation, for reconstruction.
